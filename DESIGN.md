# vivado_at_mac — Design Document

## Overview

A single CLI tool (`vam`) to run Xilinx Vivado on macOS Apple Silicon via OrbStack containers.
Edit code on Mac with VS Code, build bitstreams in a container, program FPGAs natively.

## Decision Log

| Decision | Choice | Alternatives | Reason |
|---|---|---|---|
| CLI interface | Single shell script `vam` | Makefile, docker-compose | Simplest UX, one command |
| Container runtime | OrbStack + Rosetta 2 | Docker Desktop, QEMU | Best performance, lowest RAM |
| GUI delivery | noVNC (browser) | XQuartz + X11, VNC client | Zero dependencies, reliable |
| Build method | Vivado batch mode + TCL | GUI manual operation | Fast, automatable, reproducible |
| Programming tool | openFPGALoader (Mac native) | Container hw_server | Multi-vendor, fast, no container |
| GUI programming | XVC bridge (auto) | Manual openFPGALoader | Single command UX |
| Vivado storage | `~/.vivado_at_mac/` persistent | In-container, per-project | Survives container removal |
| Project config | `fpga.yml` | Makefile, TCL files | Intuitive, minimal |
| Vivado version | 2025.2 only | Multi-version | YAGNI |

### Mount Strategy

```
Host (Mac)                          Container
─────────────────────────────────   ──────────────────
./my_project/                   →   /workspace/
~/.vivado_at_mac/Xilinx/        →   /opt/Xilinx/
~/.vivado_at_mac/.Xilinx/       →   /home/user/.Xilinx/
```

- Project files are directly accessible from both Mac and container
- Vivado installation persists across container restarts
- Container itself is stateless and disposable

## Component Design

### 1. vam CLI (shell script)

Main entry point. Subcommand dispatch:

```bash
vam <command> [args...]

Commands:
  install <bin_path>           Install Vivado from .bin file
  build [project_dir]          Build bitstream (batch mode)
  program [bit_file] [--board] Program FPGA (native openFPGALoader)
  gui [project_dir]            Open Vivado GUI in browser (noVNC)
  status                       Show installation & container status
  clean [project_dir]          Remove build/ directory
  uninstall                    Remove Vivado installation
```

Implementation notes:
- Pure bash, no external dependencies beyond docker/orbstack CLI
- Color output for status messages (green/red/yellow)
- Validates fpga.yml exists before build/gui
- Auto-detects project directory (current dir if fpga.yml exists)

### 2. Dockerfile

Base: `ubuntu:22.04` (linux/amd64)

Layers:
1. System dependencies (Vivado runtime requirements)
   - libc6-dev-i386, libgtk-3-0, libxtst6, libtinfo5, etc.
2. GUI stack (for GUI mode only, lazy-started)
   - xvfb, openbox, x11vnc, novnc, websockify
3. Utility tools
   - python3, yq (for YAML parsing), jq

NOT included in image:
- Vivado itself (mounted from host)
- Project files (mounted from host)
- Node.js, Claude Code, or other dev tools (unnecessary)

### 3. TCL Build Template (build.tcl.tpl)

Non-project mode template, variables filled from fpga.yml:

```tcl
# Auto-generated by vam — do not edit
set part    "{{PART}}"
set top     "{{TOP}}"
set outDir  "/workspace/build"

file mkdir $outDir

# Read sources
{{READ_SOURCES}}

# Read constraints
{{READ_CONSTRAINTS}}

# Synthesis
synth_design -top $top -part $part
write_checkpoint -force $outDir/post_synth.dcp
report_utilization -file $outDir/utilization.rpt

# Implementation
opt_design
place_design
phys_opt_design
route_design
write_checkpoint -force $outDir/post_route.dcp

# Reports
report_timing_summary -file $outDir/timing.rpt
report_power -file $outDir/power.rpt

# Generate bitstream
write_bitstream -force $outDir/${top}.bit
```

Non-project mode is used because:
- Faster than project mode (no intermediate project files)
- Fully reproducible (no hidden state)
- Cleaner output (only what we want in build/)

### 4. GUI Startup Script (start-gui.sh)

Runs inside container when `vam gui` is called:

```
1. Start Xvfb :1 -screen 0 1920x1080x24
2. Start openbox (window manager)
3. Start x11vnc -display :1 -nopw -forever
4. Start websockify --web /usr/share/novnc/ 6080 localhost:5900
5. Detect Vivado installation path
6. Start hw_server with XVC auto-connect to host.docker.internal:3721
7. Source Vivado settings and launch Vivado GUI
8. Wait for termination signal
```

### 5. openFPGALoader Integration

Installed on Mac host (not in container):

```bash
brew install openfpgaloader
# or use bundled binary with XVC support
```

Two modes:
- **Direct programming**: `openFPGALoader -b <board> <bitstream>`
- **XVC server**: `openFPGALoader -b <board> --xvc` (for GUI mode)

Board name comes from fpga.yml `board:` field.

## Data Flow

### Build Flow

```
fpga.yml ──parse──→ build.tcl ──vivado batch──→ build/
   │                    │                         │
   │  vam reads yaml    │  template substitution  │  outputs:
   │  extracts:         │  fills in:              │  - top.bit
   │  - part            │  - read_verilog cmds    │  - timing.rpt
   │  - top             │  - read_xdc cmds        │  - utilization.rpt
   │  - sources         │  - part/top variables   │  - power.rpt
   │  - constraints     │                         │  - checkpoints
```

### GUI Flow

```
vam gui
  │
  ├── (if board detected) start openFPGALoader --xvc in background
  │
  ├── docker run with:
  │     -v project:/workspace
  │     -v vivado:/opt/Xilinx
  │     -p 6080:6080
  │
  ├── container runs start-gui.sh
  │     → Xvfb + openbox + x11vnc + noVNC + hw_server + Vivado
  │
  └── open http://localhost:6080 in browser
```

### Program Flow

```
vam program
  │
  ├── read fpga.yml for board name and bitstream path
  │
  └── openFPGALoader -b <board> <bitstream>  (native Mac, no container)
```

## Error Handling

| Scenario | Handling |
|---|---|
| OrbStack not installed | Print install instructions, exit 1 |
| fpga.yml not found | Print error with example, exit 1 |
| fpga.yml invalid | Validate required fields, print what's missing |
| Vivado not installed | Print `vam install` instructions, exit 1 |
| Build fails (syntax error) | Show Vivado error output, preserve build.log |
| Board not connected | openFPGALoader error, suggest `--detect` |
| Port 6080 already in use | Print error, suggest `vam status` |
| Container already running | Reuse or stop existing, ask user |

## Assumptions

- User has an AMD/Xilinx account for Vivado authentication
- User has the Vivado 2025.2 Linux .bin installer downloaded
- Mac has at least 50GB free disk space
- OrbStack Rosetta 2 emulation is enabled (default on Apple Silicon)
